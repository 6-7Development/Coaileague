## AGENT 3 — Concurrency, Race Conditions & State Machines

### [CRITICAL-3A] Double-Assignment Race on Escalated Ticket Claim
- **File:** `server/routes/supportRoutes.ts:609-649`
- **Issue:** Two support agents can claim the same escalated ticket simultaneously. Read (`platformAssignedTo === null`) and write (`UPDATE ... SET platformAssignedTo = X`) are NOT in the same transaction — last write wins silently.
- **Blast radius:** Duplicate work, split chat history, customer receives two conflicting responses.
- **Fix:** Wrap in `db.transaction()` with a conditional guard:
```typescript
const [updatedTicket] = await db.transaction(async (tx) => {
  const [current] = await tx.select().from(supportTickets)
    .where(and(eq(supportTickets.id, id), eq(supportTickets.workspaceId, workspaceId!)))
    .limit(1);
  if (current?.platformAssignedTo !== null) {
    throw new Error('Ticket already claimed by another agent');
  }
  return tx.update(supportTickets)
    .set({ platformAssignedTo: staffId || userId, status: 'in_progress', updatedAt: new Date() })
    .where(eq(supportTickets.id, id))
    .returning();
});
```

### [CRITICAL-3B] No State Machine Enforcement — Closed Tickets Can Re-Open
- **File:** `server/routes/supportRoutes.ts:884-960`
- **Issue:** All status transitions are legal per `validStatuses` array. `closed → in_progress` is not blocked. `closed` should be terminal.
- **Blast radius:** Closed compliance tickets re-opened without authorization; SLA timers reset; audit trail corrupted.
- **Fix:** Add an `ALLOWED_TRANSITIONS` map and validate before writing:
```typescript
const ALLOWED_TRANSITIONS: Record<string, string[]> = {
  'open':                ['in_progress', 'waiting_for_customer', 'on_hold', 'resolved', 'closed'],
  'in_progress':         ['waiting_for_customer', 'on_hold', 'resolved', 'closed'],
  'waiting_for_customer':['in_progress', 'on_hold', 'resolved', 'closed'],
  'on_hold':             ['in_progress', 'waiting_for_customer', 'resolved', 'closed'],
  'resolved':            ['closed'],
  'closed':              [],  // TERMINAL
};
const allowed = ALLOWED_TRANSITIONS[ticket.status] ?? [];
if (!allowed.includes(status)) {
  return res.status(422).json({ message: `Illegal transition: ${ticket.status} → ${status}` });
}
```

### [HIGH-3C] Audit Log Writes Are Fire-and-Forget — SOC2 Compliance Gap
- **File:** `server/routes/shiftRoutes.ts:1196, 1246, 2696`
- **Issue:** `storage.createAuditLog({...}).catch(err => log.warn(...))` — if the write fails, the HTTP 200 still goes out and the compliance trail has a silent gap.
- **Blast radius:** SOC2 Type II audit trail is incomplete; incident forensics fail; labor dispute has no documented evidence.
- **Fix:** Audit writes must be inside the parent `db.transaction()`, or on failure, return a 507 to the client.

### [HIGH-3D] Shift Notifications Miss Idempotency Keys — Duplicate Delivery on Retry
- **File:** `server/routes/shiftRoutes.ts:917-927`
- **Issue:** `createShiftAssignedNotification()` doesn't pass `idempotencyKey` to `NotificationDeliveryService` even though the delivery table has a unique constraint on that column.
- **Blast radius:** Network retry → employee receives duplicate shift assignment notification.
- **Fix:** Generate `idempotencyKey: \`shift-${shift.id}-assigned-${shift.employeeId}\`` and include in the send call.

### [MEDIUM-3E] Escalation Notification Loop — Partial Failures Not Detected
- **File:** `server/routes/supportRoutes.ts:486-505`
- **Issue:** Notifications sent sequentially with `await` in a `for` loop — if one staff member's notify fails, the loop aborts and remaining staff are never notified.
- **Fix:** Use `Promise.allSettled()` and log failures individually.

### [MEDIUM-3F] `lockVersion` Field Exists But Not Used in Ticket Assignment
- **File:** `server/routes/supportRoutes.ts:634-642` / schema has `lockVersion` field
- **Issue:** Schema already has `lockVersion` for optimistic concurrency but the assignment endpoint ignores it.

### [LOW-3G] Ticket Closure Email Has No Idempotency Key

## AGENT 4 — Session, WebSocket & Client-State

### [CRITICAL-4A] CSRF Token Cache Outlives Session — POST Requests Fail After Idle
- **File:** `client/src/lib/csrf.ts:14-50`
- **Issue:** CSRF token cached for 30 min; session timeout is 60 min. After session expires, a cached token is sent with a dead session — server returns 403. The CSRF refresh handler then calls `/api/csrf-token` with the expired session → 401. Client is not redirected to login because the CSRF handler doesn't handle 401.
- **Blast radius:** User loses form data; no graceful logout; confusing failure mode.
- **Fix:** (1) Reduce `TOKEN_REFRESH_MS` to `< session timeout`. (2) Add `if (res.status === 401) { window.location.href = '/login'; }` inside the CSRF fetch.

### [CRITICAL-4B] WebSocket Auth Validated Only at Connect — Revoked Users Stay Connected
- **File:** `server/websocket.ts:1391-2600`
- **Issue:** `ws.serverAuth` is set once on WS upgrade and never re-checked. An admin can revoke a user's access; the user's open WebSocket continues receiving messages for hours.
- **Blast radius:** Fired employees can read live chat; compliance violation; data exfiltration vector.
- **Fix:** Re-validate session on every non-keep-alive message:
```typescript
const sessionStillValid = await storage.getSession(ws.serverAuth?.sessionId);
if (!sessionStillValid) {
  ws.send(JSON.stringify({ type: 'error', message: 'Session expired' }));
  ws.close(1008, 'Session terminated');
  return;
}
```

### [HIGH-4C] Raw `fetch()` Calls in HelpDesk Bypass Global 401 Handler
- **File:** `client/src/pages/HelpDesk.tsx:536` (MOTD), `1233` (file upload)
- **Issue:** Direct `fetch()` calls skip `apiRequest()`/`apiClient` which handles 401 by redirecting to login. Session expiry causes these calls to fail silently.
- **Blast radius:** Upload fails with no helpful message; MOTD never loads; session appears valid while broken.
- **Fix:** Replace both with `apiRequest("GET/POST", url)`.

### [HIGH-4D] WS Reconnect May Reuse Stale Auth Token (60-Sec Cache)
- **File:** `server/websocket.ts:61`
- **Issue:** WS auth tokens cached for 60 seconds. Cross-device logout within that window allows the old device to re-authenticate with a token tied to a killed session.
- **Fix:** Reduce token TTL to 10 seconds; validate session on token consumption (not just expiry time).

### [HIGH-4E] No Cross-Tab Logout Detection
- **File:** `client/src/hooks/useAuth.ts`
- **Issue:** No `StorageEvent` listener or `BroadcastChannel` for logout events. Logging out in Tab B leaves Tab A fully "authenticated" for up to 60 seconds.
- **Blast radius:** Stale tab continues making API calls; WS stays open; potential for data save conflicts.
- **Fix:**
```typescript
// On logout: localStorage.setItem('logout-event', Date.now().toString());
// In useAuth: window.addEventListener('storage', e => {
//   if (e.key === 'logout-event') { queryClient.clear(); redirect('/login'); }
// });
```

### [MEDIUM-4F] Session Timeout Warning "Stay Logged In" Does Not Extend Session
- **File:** `client/src/components/session-timeout-warning.tsx:102-111`
- **Issue:** "Stay logged in" calls `GET /api/auth/me` — this checks validity but does NOT reset the timeout. User clicks the button, feels safe, gets logged out at the original expiry time anyway.
- **Fix:** Create `POST /api/auth/extend-session` that calls `req.session.touch()` and wire the button to it.

### [MEDIUM-4G] React Query Retry May Fire on 401 — Log Spam + Minor DoS Risk
- **File:** `client/src/lib/queryClient.ts:185-201`
- **Issue:** Retry guard checks `error.status < 500` but does not explicitly exclude 401 — during logout race, multiple queries can fire 401 retries simultaneously.
- **Fix:** `if (error.status === 401 || error.status < 500) return false;`
