🧠 Enhanced Subagent Supervisor Setup
The Supervisor's primary roles are Execution, Quality Control (QC), and Context Filtering. It should handle high-volume, repetitive tasks quickly to free up the Orchestrator (Trinity AI Brain) for high-value reasoning.

1. Model Configuration
The Supervisor should primarily use the Gemini 2.5 Flash model for speed and cost-effectiveness, switching to Gemini 2.5 Pro only when a complex failure requires deep reasoning.

Role/Task	Gemini Model	Rationale
Execution Command	Gemini 2.5 Flash	Fastest model for issuing simple commands and parsing predictable JSON outputs from the Orchestrator. Low latency for high-volume execution.
Basic Quality Control (QC)	Gemini 2.5 Flash	Quick validation of Subagent output (e.g., checking if the returned JSON matches the expected schema, or if a log summary is complete).
Context Compression	Gemini 2.5 Flash	Efficiently condenses large Subagent logs or database dumps into concise, high-value summaries for the Orchestrator (saving tokens).
Failure Analysis & Escalation	Gemini 2.5 Pro	If a Subagent returns a non-recoverable error, Pro is required for deep reasoning: "Analyze this log and error type to suggest why the Storage Agent failed".

2. Core Responsibilities and Tools
A. Execution Management & Guardrails
Tool: execute_subagent_action

Action: Receives the task and arguments from the Orchestrator, logs the start time, and sends the command to the Subagent.

Tool: monitor_resource_limits

Action: Actively monitors the Subagent's execution time and memory usage. If a limit is exceeded, the Supervisor terminates the Subagent process and reports a TIMEOUT or RESOURCE_EXCEEDED failure status. (Guardrail Enforcement)

B. Quality Control (QC) & Validation
Tool: validate_schema (using Gemini Flash)

Action: After the Subagent returns, Flash quickly validates the output structure. If the output is malformed, the Supervisor reports a DATA_INTEGRITY_FAIL back to the Orchestrator.

Tool: check_compliance (using Gemini Flash and RAG)

Action: Verifies basic policy compliance (e.g., "Did the Scheduling Agent attempt to assign a shift over 12 hours?"). Uses a quick check against indexed compliance rules (RAG).

C. Context Filtering & Efficiency (Token Saving)
Tool: summarize_context (using Gemini 2.5 Flash)

Action: When a Subagent returns large data (e.g., 500 lines of debugging output), the Supervisor uses Flash to generate a concise, single-sentence summary (e.g., "The database query returned 15 results, 14 of which are viewed, matching expected count.").

Benefit: This summarized, high-value data is what is reported back to the Orchestrator, saving valuable Pro tokens and preventing context saturation.

D. Notification & Audit
Tool: update_workflow_manifest

Action: Reports the status (SUCCESS, FAILED, TIMEOUT) and latency for the specific sub-task back to the Orchestrator's central database record.

Tool: trigger_alert

Action: Immediately notifies the Universal Notification System if a high-severity error is detected, ensuring timely communication.

3. Execution Flow Example: The Perfect Sub-Task
Orchestrator (Pro) Delegate: "Supervisor, execute scheduling.find_top_employee with arguments {shift_id: 455}."

Supervisor (Flash) Executes:

Checks the credit_token_balance (essential).

Issues the command to the Scheduling Subagent.

Monitors the Subagent.

Subagent Returns: Returns a large JSON object containing 10 potential candidates.

Supervisor (Flash) Filters: Uses Flash to validate the JSON schema and identifies the top 3 candidates.

Supervisor (Flash) Reports: Sends a highly compressed summary back to the Orchestrator: "Sub-task 1 complete. Top candidates are [IDs 102, 105, 118]."

Orchestrator (Pro) Continues: Pro uses this concise data to generate the final decision (Step 2 of the workflow).
