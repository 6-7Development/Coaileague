Retrofit Plan (No Rebuilds, Only Add-Ons)

1) Keep What You Have — Add Only These Modules

Do NOT replace: chat transport, mobile UI, ticket DB, auth. Add:

1. Queue & Routing Layer (thin service)

Prioritize platform end-users automatically; route guests to general queue.

Inputs: user.role, org_id, is_subscriber, topic, SLA_tier.



2. Progress + Status Overlay (UI only)

A small, reusable component that shows: Assigned → Investigating → Waiting on you → Resolved/Escalated.

Reads ticket state from your existing system; no new DB tables.



3. Agent Toolbelt (in-chat utilities)

Buttons: Insert macro, Request file/screenshot, Send KB link, Escalate, Transfer, Create bug.

These just call your existing ticket API with new fields/labels.



4. Support “Scoped View”

A read-only impersonation mode (time-limited, logged) for SysOps to view the customer’s workspace context inside chat—no data copy.



5. AI Assist (server-side, optional)

Summaries, reply suggestions, sentiment alerts.

Runs off message events; writes internal notes back to ticket.




> Net effect: you’re wiring lightweight services and UI widgets onto the systems you already run.

2) Minimal Event Contracts (bridge chat ↔ ticket)

Keep your schema; just standardize 7 events so all parts “click”:

// 1) User enters chat
{ "type":"chat.session.started", "session_id":"...", "user_id":"...", "org_id":"...", "is_subscriber":true, "topic":"billing" }

// 2) Queue decides assignment
{ "type":"chat.agent.assigned", "session_id":"...", "agent_id":"...", "ticket_id":"..." }

// 3) Status changes (drives the progress overlay)
{ "type":"ticket.status.changed", "ticket_id":"...", "status":"investigating|waiting_user|resolved|escalated", "by":"agent|system" }

// 4) Internal note (never shown to user)
{ "type":"ticket.note.created", "ticket_id":"...", "author_id":"...", "visibility":"internal", "text":"..." }

// 5) Escalation
{ "type":"ticket.escalated", "ticket_id":"...", "from_agent":"...", "to_queue":"sysops" }

// 6) Artifact request/attach
{ "type":"ticket.artifact.requested", "ticket_id":"...", "what":"screenshot|log|file" }
{ "type":"ticket.artifact.attached",  "ticket_id":"...", "url":"..." }

// 7) Session end (close or detach)
{ "type":"chat.session.ended", "session_id":"...", "reason":"resolved|timeout|manual" }

These events are thin wrappers over your current APIs; they let you render progress, enable toolbelt actions, and drive AI assist—without new storage.

3) Roles You Already Use → Finalize Permission Map

You have: root admin, deputy admin, sysops (support). Add two light roles to cover edge cases:

moderator (optional): manage public rooms, rate-limit abusers.

Key permissions (map to your existing ACL):

chat.transfer (sysops+)

ticket.edit.status (sysops+, finance for billing topics)

ticket.note.internal (staff only)

support.impersonate.ro (time-limited, logged)

support.view.billing_scope (mask payment identifiers)


No new auth system—just add these permission keys to your current roles.

4) UX Enhancers (drop-in, low-effort)

Progress Header (shared component): reads ticket.status events; shows ETA/SLA if applicable.

Pinned Context Panel (right side on desktop, drawer on mobile): shows user/org, plan, last 3 tickets, and relevant KB links.

Macros + KB Search in composer: small popover to insert canned replies or internal KB links.

Post-close Survey + Transcript: on resolved, show 1–5 rating; email transcript (toggleable per org).


All UI reading from existing ticket endpoints—no extra backend needed.

5) Mobile Specifics (keep your current app)

Add a “Need more help?” quick-action that reopens the same ticket (status → reopened), not a new one.

If chat idle for N minutes → bot sends “Still with us?”; if no reply → set status waiting_user and keep session attachable for 24h.

Large attachments: upload direct to storage (S3/Supabase) with a pre-signed URL; chat only stores metadata.



---

6) AI Assist (opt-in, no lock-in)

Where it runs: a small service that subscribes to the events above.

What it does (safe defaults):

Summarize thread for new agent joining.

Suggest 2–3 replies (agent must click to send).

Flag “risk” (angry sentiment, legal keywords).

Auto-tag: billing, tech, how-to.


What it does NOT do: auto-send messages or change status without a human click (unless you enable it later).


This avoids breaking your current workflow while giving speed.

7) Observability (prove quality without new tables)

Add metrics + logs only:

Metrics (Prom/Grafana or simple dashboards):

FRT (first response time), AHT (handle time), CSAT, reopen rate, escalation rate, wait time by tier.


Structured logs per ticket: ticket_id, agent_id, latency_ms, action.


No schema changes; just emit logs from current services.

8) SLOs + Routing Rules (copy/paste for Replit)

SLO targets (start simple):

Priority subscribers: FRT ≤ 60s, AHT ≤ 20m, Reopen ≤ 10%

Guests: FRT ≤ 3m


Routing:

1. If is_subscriber=true → priority queue.


2. If topic = billing → finance-capable agent pool.


3. Idle > 3m → bot ping → after 10m → waiting_user.


4. resolved → survey + transcript; disconnect session.


5. escalated → silently transfer; user sees “Senior agent joined.”

PII maskers in chat (regex for SSN, card numbers).

Export/forget tool: purge a user’s transcripts on request.
